Dora vs LoRA: Parameter-Efficient Fine-Tuning for Vision-Language Models (with Vera and Qura/Qora Variants)

A concise synthesis of the latest results and practical definitions of Dora, Vera, Qura, and Qora in parameter-efficient fine-tuning.

Key high-level takeaways from the latest results

  • Dora consistently beats vanilla LoRA on a broad set of vision–language benchmarks (image–text and video–text tasks). In several tasks, Dora-based setups outperform both LoRA and full fine-tuning.
  • When scaling to visual instruction tuning with Llama-based models and CLIP ViT features (1.57B + ViT backbone), Dora delivers statistically meaningful improvements over LoRA.
  • A Vera-based direction-update variant paired with Dora (Dora + Vera) maintains or improves accuracy while using fewer trainable parameters than plain LoRA.
  • On instruction-tuning benchmarks (e.g., Alpaca-style setups), Dora adds value even when the base model is strong (Llama 7B/13B). In some configurations, Dora uses substantially fewer trainable parameters yet achieves competitive or higher scores than LoRA.
  • Parameter efficiency: a Dora variant can update only directional components for some layers (e.g., QKV in multi-head attention) and magnitude updates for others, reaching a notable reduction in trainable parameters (e.g., ~39% of LoRA’s count in a representative setup) without sacrificing accuracy.
  • Rank robustness: across ranks r ∈ {4, 8, 16, 32, 64}, Dora outperforms LoRA, with larger gains at smaller ranks—making Dora attractive for aggressive parameter budgets.

What Dora, Vera, and Qura/Qora mean (practical definitions)

  • Dora: a two-path update mechanism for each layer W ≈ (V0 + ΔV) diag(M0 + ΔM), where: - ΔV is a direction update (LoRA-style): ΔV = B @ A (low-rank factorization). - ΔM is a per-column magnitude update.
  • Vera: a LoRA-style direction-update variant used with Dora to explore alternative directional updates; Dora + Vera aims to reduce trainable parameters even further while preserving or improving accuracy.
  • Qura (quantized LoRA): a quantized/low-precision variant of LoRA intended to save memory and compute while preserving performance.
  • Qora: related quantized variant; often described as performing even better than Qura in some checkpoints. Treat Qora as part of the quantized-LoRA family; exact naming may vary by implementation.

Rank-wise takeaways

  • Across r ∈ {4, 8, 16, 32, 64}, Dora consistently outperforms LoRA on standard vision–language tasks when using Llama 7B/13B backbones.
  • The margin of improvement tends to be larger at smaller ranks (r = 4–8), which is especially appealing for aggressive parameter efficiency.

What parts get updated for parameter efficiency

  • Dora often updates a subset of transformer projection matrices (e.g., QKV projections in multi-head attention) for the directional path, while applying magnitude updates to a broader set of weight columns.
  • A representative configuration updates only the QKV directional components and uses magnitude updates elsewhere, yielding far fewer trainable parameters than LoRA while maintaining or boosting accuracy.

1) DoraLinear: Dora-style wrapper for a single linear layer


# DoraLinear: Dora-style wrapper for a single linear layer
import torch
import torch.nn as nn
import torch.nn.functional as F

class DoraLinear(nn.Module):
    """
    Dora: W ≈ (V0 + ΔV) diag(M0 + ΔM)
    - W0: frozen pre-trained weight [out_features, in_features]
    - V0, M0: decomposition of W0 (buffers)
    - ΔV = B @ A (LoRA-like direction update)
    - ΔM: per-column magnitude update
    """
    def __init__(self, linear: nn.Linear, r: int = 8, alpha: float = 1.0, dropout: float = 0.0):
        super().__init__()
        self.linear = linear
        self.r = r
        self.alpha = alpha
        self.dropout = nn.Dropout(dropout) if dropout > 0 else None

        in_features = linear.in_features
        out_features = linear.out_features

        # Freeze original weights
        self.linear.weight.requires_grad = False
        W0 = linear.weight  # [out_features, in_features]

        # Decompose W0 into V0 and M0
        eps = 1e-12
        M0 = torch.norm(W0, dim=0)  # [in_features]
        V0 = W0 / (M0.unsqueeze(0) + eps)  # [out_features, in_features]

        # Buffers for fixed part
        self.register_buffer('V0', V0)
        self.register_buffer('M0', M0)

        # LoRA-like path for ΔV: ΔV = B @ A
        self.A = nn.Parameter(torch.randn(r, in_features) * 0.01)  # [r, in_features]
        self.B = nn.Parameter(torch.zeros(out_features, r))        # [out_features, r]
        self.scaling = self.alpha / self.r

        # Magnitude path: ΔM ∈ R^{in_features}
        self.dM = nn.Parameter(torch.zeros(in_features))

        self.bias = linear.bias
        if self.dropout is not None:
            self.A = self.A
            self.B = self.B

    def forward(self, x):
        if self.dropout is not None:
            x = self.dropout(x)

        # ΔV path
        delta_V = (self.B @ self.A) * self.scaling  # [out_features, in_features]
        V = self.V0 + delta_V  # [out_features, in_features]

        # ΔM path
        M = self.M0 + self.dM  # [in_features]

        # Reconstruct W and apply
        W = V * M.unsqueeze(0)  # broadcast: [out_features, in_features]

        return F.linear(x, W, self.bias)

2) wrap attention projections with DoraLinear


# wrap attention projections in a Transformer/Vision-Language model
def wrap_with_dora(model, target_layer_names=None, r=8, alpha=1.0, dropout=0.0):
    """
    Recursively wrap Linear layers whose names match target_layer_names (e.g., q_proj, k_proj, v_proj, out_proj)
    with DoraLinear. If target_layer_names is None, wrap common attention projections heuristically.
    """
    if target_layer_names is None:
        target_layer_names = {"q_proj", "k_proj", "v_proj", "out_proj"}

    def is_target_linear(m, name):
        return isinstance(m, nn.Linear) and any(n in name for n in target_layer_names)

    from collections import deque
    queue = deque([(model, "")])
    while queue:
        cur, path = queue.popleft()
        for name, child in cur.named_children():
            full = f"{path}.{name}" if path else name
            if isinstance(child, nn.Linear) and is_target_linear(child, name):
                wrapped = DoraLinear(child, r=r, alpha=alpha, dropout=dropout)
                setattr(cur, name, wrapped)
            else:
                queue.append((child, full))
    return model

Short practitioner takeaways

  • Dora’s two-path update (ΔM and ΔV) tends to offer richer, more controllable fine-tuning signals than vanilla LoRA, especially when you want to minimize trainable parameters.
  • For vision–language benchmarks, Dora often outperforms LoRA; Vera-based directional updates can help more at low ranks.
  • In memory-constrained scenarios, Dora with lower rank or Vera/Qura-type combinations can maintain competitive accuracy with far fewer trainable parameters.
  • When deciding what to tune, focusing Dora on QKV projections in multi-head attention (with selective magnitude updates elsewhere) often yields the best accuracy-to-parameter-efficiency balance.

Short summary of the article

Dora introduces a magnitude–direction decomposition that splits updates into ΔM (per-column magnitude) and ΔV (direction) components, enabling more nuanced fine-tuning with potentially fewer trainable parameters than LoRA. New results across image–text and video–text benchmarks show Dora often outperforms LoRA, with Vera and Qura/Qora variants offering memory and efficiency advantages while maintaining competitive accuracy. Dora demonstrates rank-robust improvements over LoRA, especially at lower ranks, making it well-suited for parameter-efficient fine-tuning.
